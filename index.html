Vagrant

“Vagrant is a tool for building complete development environments. With an easy-to-use workflow and focus on automation.”

Vagrant

  • Started in January 2010 by Mitchell Hashimoto.

  • Manage virtual environments on an automated way.

  • It uses standard technologies:

    • Supports VirtualBox, VMware, AWS...

    • Provisioning available with shell scripts, Chef y Puppet.

  • Getting very popular among Devs, Sysadmin and DevOps.

  • Open source project. MIT license.

  • Good documentation.

  • Great support from community.

Vargant
Vagrant

For Developers:

  • Allows you to have consistent environments (using VMs).

  • Environments isolated from the host machine. Nothing that is installed on your (real) computer could be affected (browsers, IDEs, etc.).

  • Independent from the OS you have installed on the host machine (well, almost).

  • Everyone on a team share the same settings on the environment for a project.

  • Manage multiple environments at same time.

  • Almost all interaction is done through the command-line interface.

  • Portable environments.

Vagrant

For SysAdmins/DevOps:

  • Allows you to have consistent environments (using VMs).

  • Wide set of options for tools: shell scripts, Chef cookboks, VM and more (local servers, AWS).

  • A great tool to organize and unifiy work.

  • Can be used to make tests or do experimental stuff.

  • Did I said interaction through the CLI?

Vagrant

For Visual Designers, UI devs and other roles:

  • Allows them to be abstracted from the environment setup.

  • Decrease the impact each time config/settings change are required on the env.

Vagrant

Advantages:

  • Customizable.

  • Simple commands set.

  • Allows you to same environment (as possible) than Production env.

  • Allows you to have different envs. per project, with its own OS/settings.

  • Great options to choose for provisioner.

  • None conflict with sistems VCS lice Git, Mercurial, SVN.

  • Re-usable and portable environments.

Vagrant

Disadvantages:

  • Use of VM => less resources for host machine.

  • Compatible with several platforms, but has some limitations (NFS with Microsoft)

  • It forces the Dev/Sysadmin to learn about one of the provisioners.

  • On teams (specially with devs) people with less/none experience on setup envs, have less chance to learn about this. (if they are not curious)

Vagrant

Some concepts:

  • Box: package format for Vagrant environments.

    • vagrant box add precise32 http://files.vagrantup.com/precise32.box

  • Synced folders: enables Vagrant to sync a folder on the host machine to the guest machine, by default: /vagrant

  • Provisioning: allow you to automatically install software, alter configurations, and more on the machine.

Vagrant

Vagrant commands:

  • vagrant init: initializes current directory as a Vagrant environment, creating an initial Vagrantfile if one doesn't already exist.

  • vagrant up: creates and configures guest machines according to the Vagrantfile.

  • vagrant provision: runs any configured provisioners against the guest machine.

  • vagrant halt: shuts down the running machine.

  • vagrant reload: same as: vagrant halt + vagrant up

  • vagrant ssh: ssh to the running machine.

Vagrant

Typical command workflow:

  • vagrant init: in case of start on proyect.

  • vagrant up: to create & config guest machine or to turn on the machine.

  • vagrant provision: to run all or a specific provisioner.

  • vagrant ssh: for remote access to guest machince.

  • ...

  • Work as usual on the environment.

  • ...

  • vagrant halt: turn of the machine.
